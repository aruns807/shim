Contributing to open source is hard.

Contributing to open source is a great learning experience but,

  • Docs aren't always clear
  • How one should go about contributing isn't obvious
  • Codebases are gigantic and intimidating...etc.

See: 

https://bugzilla.mozilla.org/buglist.cgi?quicksearch=sw:[good%20first%20bug]

Vim could be better

Vim is awesome. But:

  • The learning curve is ridiculous
  • Plugins could be better (Install a plugin to manage your plugins? What?)
  • It has tons of features that most people don't use (refer to point 1)

Let's make both better!

No, let's not rewrite Vim. Instead, let's make a text editor that:

  • Is modal like vim
  • Is highly configurable like Vim
  • Understands a subset of commonly used vim motions (dw, m[buffer name], etc).
  • Has a plugin system that behaves like "real" code (none of that API/scripting stuff)

Doing so will give people a project that they can both use day to day as well as contribute to (something that I wish I had when I first started with open source).

Plugins are injected directly into the codebase (via a loading script). That way plugins have access to every single state variable in the program. Code specific to each plugin can be segmented into it's own module to prevent naming collisions.

So what next?

Unfortunately shim isn't ready to be used yet (I've been working at this for the past week or so and the RSI is starting to kick in).

There is still tons of work to be done! Create a proper unit testing suite, create proper plugin loading logic, tons of design decisions to be made... yada yada. As of right now the repo is just slightly under two thousand lines and there is tons of foundational stuff that needs to be done so contributing should be extremely straightforward.
